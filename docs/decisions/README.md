# Decision System — savdo-builder

**Версия:** 1.0
**Дата:** 2026-05-20
**Назначение:** единая система принятия командных решений по числам, а не «на ощупь».

---

## Когда применять какой инструмент

| Ситуация | Инструмент | Файл |
|----------|------------|------|
| Не понятно, нужно ли вообще формальное решение | Methodology | [`framework.md`](framework.md) |
| Выбираем 1 из N (бренд-имя, библиотека, провайдер платежей) | Scoring matrix | [`templates/scoring-matrix.md`](templates/scoring-matrix.md) |
| Принятое архитектурное решение нужно зафиксировать | ADR | [`templates/adr.md`](templates/adr.md) → копировать в [`../adr/`](../adr/) |
| Готовы ли мы запускать в прод | Go/No-Go checklist | [`templates/go-no-go-checklist.md`](templates/go-no-go-checklist.md) |
| Сравниваем технические опции (X vs Y vs Z библиотеки) | Feature comparison | [`templates/comparison-feature-options.md`](templates/comparison-feature-options.md) |
| Конкретное живое решение | Применённый документ | `<topic>-YYYY-MM-DD.md` (см. ниже) |

## Pipeline: «есть вопрос → принято решение»

```
1. Вопрос                  →  framework.md → нужно ли формальное решение?
2. Если ДА                 →  выбрать шаблон (scoring / comparison / go-no-go)
3. Заполнить копию шаблона →  /docs/decisions/<topic>-YYYY-MM-DD.md
4. Получить решение        →  записать ADR в /docs/adr/ADR-NNN_<slug>.md
5. Поставить review-точку  →  Obsidian TODO (1 / 3 / 6 месяцев)
```

Без шагов 4 (ADR) и 5 (review-точка) решение **не считается принятым** — оно «висит в воздухе» и через 2 месяца никто не помнит, почему так.

## Правила гигиены

1. **Один файл = одно решение.** Не смешивать «выбор имени» и «выбор провайдера платежей» в одном документе.
2. **Веса критериев публикуются ДО оценки альтернатив.** Иначе подгоняем веса под желаемый ответ.
3. **Оценщиков ≥ 2.** Полат + Азим выставляют независимо, потом усредняют. Расхождение > 3 баллов → обсудить, не усреднять.
4. **Score < 6.0 — не победитель.** Если даже топ-альтернатива < 6/10 — критерии или варианты выбраны плохо, ищите ещё.
5. **Решение в ADR пишется в день принятия.** Не «потом задокументируем».

## Что НЕ требует формального решения

- Внутренние имена переменных, файлов (если соответствуют конвенциям проекта).
- Choice между двумя equivalent-библиотеками одного класса (lodash vs ramda) если разница < 10% по любому критерию.
- Bug-фикс — не решение, а fix. ADR не нужен.
- Cosmetic UI-tweaks внутри согласованной дизайн-системы.

См. полный список «не нужно фиксировать» в [`framework.md` §3](framework.md#3-когда-формальное-решение-не-нужно).
