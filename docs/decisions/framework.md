# Decision Framework — методология

**Версия:** 1.0
**Дата:** 2026-05-20
**Применимо к:** savdo-builder (api / admin / web-buyer / web-seller / mobile-*)

---

## 1. Зачем нужна формальная система

В команде из 3 человек (Полат, Азим + опционально контракторы) большинство решений можно принять «на чате». Но три класса решений ВСЕГДА должны быть формальными:

- **Architectural** — затрагивает > 1 модуль или меняет инвариант (`docs/V1.1/`).
- **Brand / Product** — публичные атрибуты (имя, логотип, палитра, voice).
- **Go / No-Go** — запуск, миграция данных, breaking change в API.
- **Vendor / Tooling** — выбор провайдера ($, lock-in, миграция-стоимость).

Цена ошибки в этих классах = недели работы. Поэтому — структура.

---

## 2. Когда формальное решение НУЖНО

| Признак | Пример из savdo-builder |
|---------|------------------------|
| Стоимость отката > 3 дней работы | Замена ORM, смена платёжного провайдера |
| Затрагивает > 1 домен (api+web+admin) | Структура `Order.status` enum |
| Меняет public API (web/mobile/bot контракт) | Переименование `/v1/orders/:id/cancel` |
| Влияет на users / sellers (UX) | Имя бренда, основной CTA-цвет |
| Безвозвратные изменения в БД | DROP column, миграция данных |
| Решение с финансовым impact-ом > $50/мес | Платный observability tool, paid SaaS |
| Решение, которое кто-то через 6 месяцев спросит «почему так» | Любой из выше |

→ Применить scoring matrix + ADR + review-точку.

---

## 3. Когда формальное решение НЕ нужно

- Routine code: переименовать private function, вынести helper, dedupe utility.
- Bugfix без change в архитектуре (`fix N+1 on /products` — это PR, не ADR).
- Choice между двумя equivalent options, оба нативно в стеке (e.g. `Array.map` vs `for...of`).
- Cosmetic UI-tweaks внутри уже согласованной дизайн-системы (`docs/brand/`).
- Внутренний рефакторинг при сохранении публичных контрактов.
- Любое решение, которое можно откатить за < 30 минут без потери данных.

→ Просто сделать. PR review достаточно.

## 4. Pipeline (7 шагов)

### Шаг 1 — Формулировка проблемы

Один абзац, без жаргона. Шаблон:

> «Нам нужно [X], потому что [Y]. Текущее состояние: [Z]. Если не решим — [последствия].»

Пример:
> «Нам нужно выбрать платёжного провайдера для UZ, потому что без онлайн-оплаты конверсия в TG-флоу падает на ~40%. Сейчас оплата manual через chat. Если не решим до launch — потеряем sellers, которым нужны автозакрытия.»

**Anti-pattern:** «Нужна интеграция с Click». Это уже ответ, а не проблема.

---

### Шаг 2 — Критерии

3–8 критериев. Не больше — иначе все веса размажутся и победит «средний» вариант. Не меньше — иначе scoring превращается в «нравится / не нравится».

Каждый критерий — измеримый или хотя бы оценочно ранжируемый 0–10.

Примеры критериев для **бренда:**
- Memorability — запоминается ли с первого раза
- UZ-fit — резонирует ли с локальным контекстом
- Pronunciation — легко ли произнести по-русски и по-узбекски
- Domain availability — свободны ли `.uz`, `.com`

Примеры критериев для **технического выбора (платёжный провайдер):**
- Покрытие банков UZ (Humo, UzCard, Visa, MC)
- Комиссия (% + fixed)
- Время интеграции (developer-days)
- SLA + поддержка
- Compliance (PCI, локальные требования)

**Anti-pattern:** «качество» как критерий. Что такое качество — UX, performance, code coverage? Декомпозируй.

---

### Шаг 3 — Веса

Сумма весов = **1.0**. Каждый вес в диапазоне 0.05–0.30.

- 0.05 — минимальный, «приятный бонус»
- 0.10 — стандартный второстепенный
- 0.15 — важный
- 0.20 — критичный
- 0.25–0.30 — must-have, если критерий проваливается — решение нельзя принять

**Правило:** веса публикуются **до** оценки альтернатив. Иначе подсознательно подгоняем веса под желаемый ответ.

**Anti-pattern «равные веса 1/N»** — означает что мы не подумали о приоритетах.

---

### Шаг 4 — Альтернативы

Минимум 3 варианта. Если 2 — добавь «status quo / не делать ничего» как третий. Если 1 — это не выбор, это план.

Каждая альтернатива описывается одинаковым уровнем детализации (нельзя «Click — крупный известный провайдер» против «Payme — белорусская компания, основана в 2012, входит в группу IMAN, поддерживает X банков...»). Иначе scoring смещается в пользу детально описанной.

---

### Шаг 5 — Scoring

Каждый оценщик выставляет 0–10 по каждой паре (альтернатива × критерий) **независимо**.

- 0–2 — провал, неприемлемо
- 3–4 — плохо, но допустимо
- 5–6 — средне
- 7–8 — хорошо
- 9–10 — превосходно

Расхождение между оценщиками > 3 балла → обсуждение, не среднее арифметическое. Часто там скрытый критерий, который один видит, а другой нет.

Финальный взвешенный балл:
```
score(alt) = Σ (weight_i × rating_i)
```

---

### Шаг 6 — Решение

- Победитель = max(score), при условии score ≥ 6.0.
- Разрыв с #2 < 0.5 балла → пересмотреть критерии или принять решение волевым порядком (с записью в ADR).
- Если ни один не ≥ 6.0 → reset, варианты слабые.

---

### Шаг 7 — ADR + review-точка

ADR (см. [`templates/adr.md`](templates/adr.md)) пишется **в день принятия решения**. Файл: `docs/adr/ADR-NNN_<slug>.md`.

Review-точка — Obsidian TODO с датой:
- Architectural — через 6 месяцев
- Brand — через 12 месяцев
- Vendor — через 6 месяцев + при любом инциденте
- Go/No-Go — через 30 дней после launch

Команда:
```powershell
pwsh obsidian-note.ps1 todo savdo-builder "Review ADR-NNN" "Дата: <date+N>. Проверить — гипотеза подтвердилась?"
```

## 5. Anti-patterns (запрещены)

### 5.1 Sunk cost fallacy

«Мы уже потратили 2 недели на интеграцию X, давайте не менять, хоть Y явно лучше».

**Контрмера:** в scoring matrix у критерия «реализованность сейчас» вес ≤ 0.10. Если он перевешивает — мы не считаем, а оправдываемся.

### 5.2 Recency bias

«Вчера читал статью про rust — давайте перепишем backend». Свежее впечатление переоценивает плюсы новинки и недооценивает её риски.

**Контрмера:** новые технологии < 6 месяцев на рынке получают штраф −2 балла к критерию «зрелость / community».

### 5.3 HiPPO (Highest Paid Person's Opinion)

«Я СТО сказал X». В команде из 3 это особенно опасно, потому что нет противовеса.

**Контрмера:** scoring заполняется независимо. Финальные числа — единственный аргумент. Если кто-то хочет переопределить — пишет volevoe решение в ADR с явным «волевое: $person, причина: $reason».

### 5.4 Bikeshedding (закон тривиальности)

Час спорим о цвете кнопки, 5 минут о выборе ORM. Простые вопросы получают много внимания, сложные — мало.

**Контрмера:** временной таймбокс — обсуждение scoring matrix не больше 60 минут. Дальше — записываем варианты, расходимся, оцениваем независимо.

### 5.5 Bandwagon / hype-driven

«Все используют Next.js App Router, поэтому мы тоже». Может быть прав, может нет — но это не аргумент.

**Контрмера:** «популярность» как критерий имеет вес ≤ 0.05. Если технология популярна — она получит высокий балл по «community / docs», этого достаточно.

### 5.6 Premature optimization

«Что если у нас будет 1M sellers — выберем архитектуру под это». Сейчас 0 sellers.

**Контрмера:** критерий «scalability» оценивается на горизонт 12 месяцев + 10× от текущей нагрузки, не «теоретически до infinity».

### 5.7 Analysis paralysis

Месяц выбираем библиотеку логирования. Цена решения < цены задержки.

**Контрмера:** для решений с откатом < 1 неделя — таймбокс 2 часа на scoring, дальше — fail forward.

### 5.8 Anchoring

Первая предложенная цифра (бюджет, deadline, оценка) задаёт «якорь» и все последующие оценки крутятся вокруг.

**Контрмера:** оценщики выставляют 0–10 не зная оценок коллег. Только после — сверка.

## 6. Review-точка: проверка решения постфактум

Через N месяцев (см. шаг 7) — проверяем гипотезу.

Шаблон проверки:

```markdown
## Review ADR-NNN (дата +N месяцев)

- **Гипотеза была:** ...
- **Фактически:** ...
- **Подтверждено / опровергнуто:** ✅ / ❌ / частично
- **Дельта (если опровергнуто):** что мы упустили в критериях / весах?
- **Действие:** keep / migrate / re-decide
```

Невыполненный review = decision rot. Через 18 месяцев никто не помнит, почему так.

---

## 7. Связь с другими процессами

- **Tasks в работу** — после ADR → создать задачу в `analiz/tasks.md` с ссылкой на ADR.
- **Bugs** — root cause найден → ADR может потребоваться, если фикс затрагивает архитектуру.
- **Obsidian** — каждое формальное решение → `obsidian-note.ps1 decision savdo-builder ...`.
- **Brand book** — изменения terracotta / wordmark / voice → отдельный ADR + обновление `docs/brand/`.
