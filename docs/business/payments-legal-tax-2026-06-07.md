## 1. Платёжная модель — РЕШЕНО

### 1.1 Деньги покупатель → продавец (оплата заказа): мы НЕ трогаем
- maxsavdo — **чистый SaaS-инструмент**, не маркетплейс и не платёжка. Деньги сделки идут **напрямую** продавец↔покупатель и **через нас не проходят**.
- На checkout мы **только показываем** способы оплаты продавца. Это «Вариант A».
- Следствие: ноль финансовой ответственности (нет возвратов, споров, chargeback, escrow, KYC под выплаты), не нужен статус платёжного агента / эквайринг под покупательские деньги.

### 1.2 Комиссии с оборота (GMV) — НЕ берём
- Ранее в `business-model-v2` §0 был заложен **Phase 2 = hybrid (подписка + комиссия 1–3% с оборота)**. **Это решение отменено (07.06.2026).**
- Единственный источник денег maxsavdo — **подписка продавца**. Комиссии с GMV нет.
- Бонус позиционирования: «**0% комиссии**, в отличие от Uzum (10–25%)» становится железобетонным аргументом.
- TODO: внести правку в `business-model-v2` §0/§7 (демоут Phase 2 hybrid → «решено: subscription-only»).

### 1.3 Способы оплаты заказа на checkout (под реалии УЗ)
Доставка в УЗ обычно — **такси**, самовывоз или встреча (НЕ «курьер с POS-терминалом»). Передача налички — как продавец с покупателем сами договорятся. Платформа механику передачи не моделирует, только показывает способ:
- **Наличные** — по договорённости (без слова «курьер»).
- **Перевод на карту** — показываем карту продавца из поля реквизитов + «скопировать».

**⚠️ Баг текущего кода:** в `apps/web-buyer` checkout захардкожено «Наличные курьеру» / «Картой курьеру — UzCard/Humo POS-терминал» — это фикция для мелкого TG-продавца. Переделать под список выше; «Online (Payme/Click)» убрать/оставить «Скоро» (в эквайринг покупательских денег не идём).

### 1.4 Реквизиты продавца — как карта попадает к покупателю
- Выбран путь: **поле реквизитов в кабинете продавца** (не «кидать карту в чат»).
- Продавец один раз вписывает: номер карты (8600…), имя владельца, (опц.) Click/Payme-ссылку, отмечает принимаемые способы.
- Покупатель на checkout видит **только то, что принимает этот продавец**.
- **Зоны:** поле карты в `Seller`/`Store` = `packages/db` (**Полат** — нужен запрос). Ввод (web-seller) и показ (web-buyer) = **Азим**.

### 1.5 Поток оплаты заказа (итог)
1. Продавец (один раз): кабинет → реквизиты → карта + способы.
2. Покупатель: checkout → видит способы продавца → «перевод» → карта + «скопировать».
3. Покупатель переводит сам, (опц.) «Я оплатил».
4. Продавец сверяет приход, отмечает «оплачено». Деньги мимо нас.

## 2. Подписка продавца → maxsavdo

### 2.1 Что уже построено (Полат)
Схема готова под оба этапа: `Subscription` (TRIAL→ACTIVE→PAST_DUE→SUSPENDED→CHURNED/CANCELLED, триал, period, grace, скидки beta/migration/referral), `SubscriptionPayment` (`MANUAL_TRANSFER` сейчас; `CLICK`/`PAYME`/`COMP` + `externalId` под Phase 2), admin подтверждает оплату вручную.

### 2.2 Чего нет
В кабинете продавца (web-seller) **нет экрана покупки подписки** — есть только публичная `Pricing.tsx` на лендинге. UI «выбрать тариф и оплатить» — зона Азима, не построен.

### 2.3 «Онлайн-подписка» — что это требует
Настоящий онлайн (нажал → списалось → активна без человека) = **юрлицо + расчётный счёт + мерчант Payme/Click + webhook**. До этого — только перевод на личную карту с ручным подтверждением (серо на масштабе). **Главный анлок онлайн-подписки = регистрация + касса (раздел 3).**

### 2.4 Этапность (UI строится один раз, движок оплаты под ним меняется)
- **Этап 1 (пока нет кассы):** кабинет → «Тарифы» → deeplink Payme/Click + «Я оплатил» → `SubscriptionPayment(PENDING)` → админ подтверждает → `ACTIVE`. Для беты (10–30 продавцов) ручное подтверждение — норм.
- **Этап 2 (касса готова):** та же кнопка → реальный виджет → webhook → автоактивация. Экран продавца не меняется.
- Решение «запускать ли Этап 1 сейчас» — **отложено**: сначала бизнес-модель «чётко», потом стройка.

## 3. Юридическое оформление и налоги (УЗ) — план действий

### 3.1 Главный вывод
Оптимальный путь для SaaS — **юрлицо (ООО) + резидентство IT Park**. IT Park почти обнуляет налоги для софта — легальная защита от налоговых проблем. **Личную карту для приёма подписок на масштабе — нельзя** (риск №1).

### 3.2 Выбор формы

| Форма | Налог (2026) | Payme/Click | IT Park | Вердикт |
|---|---|---|---|---|
| Самозанятый | 1% оборота (до 1 млрд сум) | ❌ | ❌ | Не годится (нельзя касса, нельзя IT Park; список видов урезан 104→72). |
| ИП | 1% оборота (до 1 млрд сум) | ✅ | ⚠️ с оговорками | Быстрый старт; минус — личная ответственность, тяжелее с соучредителями/инвесторами. |
| **ООО (MChJ)** | 1% оборота / IT Park → ~0% | ✅ | ✅ (без требований к уставному фонду для местных) | **Рекомендуется.** Лимит. ответственность, путь к IT Park и инвестициям. |

### 3.3 IT Park — почему критично
Резиденты: **0% налога на прибыль, 0% НДС, 0% соцналога, 0% налога с оборота** (базово до 01.01.2028) + зарплатный налог сотрудников **7.5%** (вместо 12%). Условия: юрлицо в УЗ + бизнес-план + IT-деятельность из перечня (SaaS подходит) + рабочие места местным.
⚠️ Супер-льготы до **2040** + 5% на дивиденды — для **экспортёров** (>50% выручки из-за рубежа). У нас выручка внутренняя → **базовый пакет до 2028** (всё равно ~0%), не экспортный трек.

### 3.4 Как легально принимать подписку
1. Расчётный счёт в банке.
2. Договор с **Payme Business** и/или **Click** (мерчант). Онлайн-договор — через **didox** с ЭЦП. Получаешь **Merchant ID + Secret Key** → это и есть «Phase 2 webhook».
3. **ЭСФ (электронные счета-фактуры)** продавцам + **публичная оферта** (`offer/page.tsx`, задача `LEGAL-OFFER-REQUISITES-001`).
4. Вся выручка — **только через расчётный счёт**, не через личную карту.

### 3.5 Налоги (2026)
- До 1 млрд сум/год: **1% с оборота** (или ~0% в IT Park).
- Свыше 1 млрд сум: общий режим — **НДС 12%** + налог на прибыль 15% (первый год прибыль 0% при первом переходе; в IT Park — освобождение).
- Бухгалтерия — **обязательно аутсорс-бухгалтер**.

### 3.6 Чек-лист Азиму (бизнес-задачи, не код)
1. ☐ Выбрать форму (рекоменд. **ООО**) — согласовать с соучредителем, если есть.
2. ☐ Получить **ЭЦП**.
3. ☐ Зарегистрировать юрлицо (one-stop birdarcha / davlat xizmatlari).
4. ☐ Открыть **расчётный счёт**.
5. ☐ Подать на **резидентство IT Park** (бизнес-план + IT-ОКЭД).
6. ☐ Договор с **Payme Business** / **Click** → Merchant ID.
7. ☐ Нанять **аутсорс-бухгалтера**.
8. ☐ Заполнить реквизиты в **публичной оферте** (`LEGAL-OFFER-REQUISITES-001`).

Пункты 4–6 = настоящая касса = анлок онлайн-подписки (§2.3).
