## 1. Цель и контекст

Исходный тикет `LANDING-CORP-PAGE-001` в `analiz/tasks.md` просил отдельную
корпоративную страницу `/about` в `apps/web-buyer` с CTA на три аудитории
(покупатель/продавец/админ) + контакты. Он был написан до того, как появился
маркетинговый лендинг на `maxsavdo.uz`.

**Факт на 11.07.2026** (проверено по коду, не по `tasks.md`): полноценный
seller-маркетинговый лендинг уже существует и задеплоен — `Hero`, `SocialProof`,
`ProblemSection`, `HowItWorks`, `Showcase`, `WhyUs`, `Features`, `Pricing`, `Faq`,
`FinalCta`, `LandingHeader`, `LandingFooter` (`apps/web-seller/src/components/landing/`).
Он живёт **не в `main`**, а в двух deploy-ветках:

- `landing` → Railway-сервис `landing`, домен `maxsavdo.uz`
- `web-seller` → Railway-сервис `savdo-builder-sl`, домен `seller.maxsavdo.uz`

Обе ветки сейчас идентичны по этим файлам (синхронизированы фиксом
`LANDING-BRANCH-DRIFT-001`, 08.07.2026). `main` не деплоится ни на один из этих
доменов — его `apps/web-seller/src/app/page.tsx` это просто `redirect('/dashboard')`,
расходящийся с обеими прод-ветками (отдельная проблема, вне scope — см. §6).

Лендинг весь заточен под конверсию продавца (единственный CTA в Hero — «Создать
бесплатно» → `/login`, плюс demo-ссылка на один конкретный магазин). У покупателя,
который просто хочет посмотреть каталог магазинов, и у команды, которой нужна
техническая ссылка на админку, — нет входа.

**Новая формулировка задачи:** не строить страницу с нуля, а добавить
недостающие точки входа в уже существующий лендинг, не трогая его
seller-conversion воронку (Hero остаётся как есть).

### Не входит в этот спек
- Email/support-канал контакты — заблокированы `LEGAL-OFFER-REQUISITES-001`
  (юр.лицо не зарегистрировано) и `SUPPORT-CHANNEL-001` (канал не создан
  Полатом). Добавлять их сейчас значит писать несуществующие контакты.
- Правки Hero / seller-конверсии.
- Синхронизация `main` с прод-ветками (см. §6, отдельный follow-up).
- Собственно домен админки — берём его через env var с фолбэком (см. §4),
  подтверждение точного значения — за Полатом.

## 2. Выбор подхода к веткам

| Вариант | Плюсы | Минусы |
|---|---|---|
| Править на `main`, ждать отдельного деплоя | Чисто с точки зрения git-истории | На `main` этого кода вообще нет (Hero/LandingPage отсутствуют) — пришлось бы портировать весь лендинг. Не относится к задаче. |
| Править сразу в обеих ветках вручную (checkout по очереди в рабочем дереве) | Просто | Рабочее дерево сейчас на `main` с только что закоммиченными, но ещё не запушенными правками онбординга — переключение веток без изоляции рискованно. |
| **Git worktree на `landing`, затем cherry-pick коммита на `web-seller`** ✅ | Не трогает текущее рабочее дерево/`main`; соответствует правилу проекта (`CLAUDE.md`: «Используй `.worktrees/` для git worktrees»); ветки идентичны по целевым файлам → чистый cherry-pick без конфликтов | Два пуша вместо одного — приемлемо, ветки и так независимые deploy-таргеты |

**Выбран** третий вариант.

## 3. Изменения в коде

Все пути — относительно `apps/web-seller/src/`, применяются в worktree на
`landing`, затем тем же коммитом cherry-pick на `web-seller`.

### `components/landing/LandingHeader.tsx`
- Десктоп-нав (между `ANCHORS` и блоком auth-CTA) и мобильное меню: текстовая
  ссылка на `buyerOrigin()` (внешняя, `target="_blank" rel="noopener noreferrer"`),
  подпись — ключ `nav.buyerCatalog`.
- Клик трекается тем же паттерном, что остальные CTA:
  `landingTrack('landing_cta_clicked', { place: 'header-buyer-catalog' })`.
- Импорт `buyerOrigin` из `@/lib/buyer-url` (уже используется в `LandingFooter.tsx`
  и `Hero.tsx` → `buyerStoreUrl`, паттерн обкатан).

### `components/landing/LandingFooter.tsx`
- Сетка `grid-cols-3` → `sm:grid-cols-4`.
- Новая 4-я колонка с заголовком `footer.product` (ключ уже существует в
  `ru.ts`/`uz.ts`, сейчас нигде не используется — очевидно задумывался для
  этого):
  - ссылка «Каталог магазинов» (`nav.buyerCatalog`, переиспользуем ключ) →
    `buyerOrigin()`
  - мелкая приглушённая (`colors.textDim`, меньший `text-[11px]`, без акцентного
    цвета — не промо) ссылка `footer.admin` → `NEXT_PUBLIC_ADMIN_URL`

### `lib/env.ts` / инлайн в компоненте (следуя паттерну `BOT_USERNAME` в `OtpGate.tsx`/`products/[id]/page.tsx`)
```ts
const ADMIN_URL = process.env.NEXT_PUBLIC_ADMIN_URL ?? 'https://adminsb.up.railway.app';
```
Фолбэк — самый свежий URL админки, встреченный в security-аудитах Полата
(`analiz/logs.md`, май-июнь 2026). В `analiz/tasks.md` есть более старое
значение (`savdo-builderadmin-production...`, апрель) — расхождение подтверждает
смену домена со временем, поэтому решение через env var, а не хардкод: если
Полат подтвердит другое значение, правится один env var в Railway, без деплоя
кода.

### i18n — оба файла (`ru.ts`, `uz.ts`), обе ветки
```ts
// ru.ts
'nav.buyerCatalog': 'Каталог магазинов',
'footer.admin': 'Админка',

// uz.ts
'nav.buyerCatalog': "Doʻkonlar katalogi",
'footer.admin': 'Administratsiya',
```

## 5. Проверка

- `tsc --noEmit` на worktree (`landing`) после правок.
- Визуально: `pnpm --filter web-seller dev` в worktree, скриншот `/` (десктоп +
  мобильное меню, RU и UZ через `LangToggle`).
- Убедиться что `buyerOrigin()` в проде резолвится в `NEXT_PUBLIC_BUYER_URL`
  (сейчас `shop.maxsavdo.uz` по фолбэку в `lib/buyer-url.ts`) — открыть ссылку
  из хедера, проверить что ведёт на реальный каталог, не 404.
- После cherry-pick на `web-seller` — тот же визуальный чек на этой ветке
  (кабинет продавца не должен пострадать, `/login`/`/dashboard` не трогаются).

## 6. Риски / follow-up (не в этом спеке)

- **`main` расходится с прод-ветками лендинга** (`main`: redirect-заглушка,
  `landing`/`web-seller`: полный маркетинг-лендинг). Не первый раз всплывает
  этот класс проблем (`LANDING-BRANCH-DRIFT-001`). Стоит завести отдельный
  тикет на объединение — либо смёржить лендинг-код в `main` и деплоить оттуда,
  либо явно задокументировать в `CLAUDE.md`, что `landing`/`web-seller` —
  самостоятельные deploy-ветки не для мержа. Решение — не моё (архитектурное,
  затрагивает Railway-конфиг Полата).
- **`NEXT_PUBLIC_ADMIN_URL` фолбэк не подтверждён Полатом** — подстраховано
  env-var подходом (см. §3), но стоит явно спросить точный текущий домен
  админки при следующем контакте.
