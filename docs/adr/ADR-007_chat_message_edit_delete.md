# ADR-007 — Редактирование и удаление сообщений в чате

**Дата:** 2026-05-15
**Статус:** Accepted (задним числом — фича внедрена 2026-04-26, session 36)
**Авторы:** Полат, Азим
**Связано с:** [ADR-004](ADR-004_chat_scope.md), INV-CH02

## Контекст

[ADR-004](ADR-004_chat_scope.md) (2026-03-21) явно исключил «Delete / Edit
messages» из MVP-scope чата. Инвариант **INV-CH02** в
`docs/V1.1/01_domain_invariants.md` гласит: «Chat messages — append-only в MVP.
Удаление и редактирование не поддерживаются. `chat_messages.is_deleted`
зарезервировано для будущего».

Однако в session 36 (2026-04-26) edit/delete сообщений был **реализован**:
- Backend: `chat.controller.ts` — `editMessage` / `deleteMessage` use-cases
- TMA + web-buyer + web-seller: `⋮` меню сообщения с «Изменить» / «Удалить»
- WS-события `chat:message:edited` / `chat:message:deleted`

Это сделано совместным решением Полата и Азима, но **без ADR** — формально
выглядит как scope creep (нарушение ADR-004 + INV-CH02). Web-sync аудит
14.05.2026 (`analiz/audits/web-sync-2026-05-14.md`) флагнул это как
недокументированное отклонение. Данный ADR закрывает пробел.

## Решение

**Edit / delete сообщений включены в scope** со следующими ограничениями.

### Edit (редактирование)

- Доступно **только автору** сообщения.
- **Время-окно: 15 минут** с момента отправки (`EDIT_WINDOW_MS` во фронте,
  enforced также на backend в `editMessage` use-case).
- Только text-сообщения (не фото).
- После редактирования сообщение получает `editedAt: timestamp`.
- UI помечает изменённое сообщение лейблом «· изменено» / «· tahrirlangan».
- WS-событие `chat:message:edited` — все участники thread'а видят правку realtime.

### Delete (удаление)

- **Soft-delete only.** `chat_messages.is_deleted = true`, текст обнуляется.
- Доступно **только автору** сообщения. Без time-окна (можно удалить старое).
- Запись в БД сохраняется (для audit / counter-консистентности) — `is_deleted`
  скрывает контент, не строку.
- UI рендерит placeholder «Сообщение удалено» вместо контента.
- WS-событие `chat:message:deleted`.

### Что НЕ меняется

- Append-only характер таблицы сохраняется физически — `DELETE FROM` не
  выполняется, только `UPDATE is_deleted`.
- Order-thread сообщения после доставки/отмены — read-only (нельзя edit/delete
  в закрытом order-thread).
- `is_deleted` поле — ровно то что ADR-004 «зарезервировал для будущего».
  Будущее наступило.

## Обновление INV-CH02

`docs/V1.1/01_domain_invariants.md` INV-CH02 переформулирован:

> **INV-CH02** — Chat messages — soft-mutable в пределах ограничений ADR-007.
> Физически append-only (нет hard-delete). Edit — только автор, 15-мин окно.
> Delete — только автор, soft (`is_deleted`).

## Последствия

**Плюсы:**
- UX-паритет с привычными мессенджерами (Telegram сам позволяет edit/delete)
- Покупатель может исправить опечатку в адресе / контакте
- Soft-delete сохраняет данные для возможного review модерацией

**Минусы / риски:**
- 15-мин окно edit — компромисс. Слишком короткое — пользователь не успеет;
  слишком длинное — можно «переписать историю» спора. 15 мин выбрано
  эмпирически (хватает на опечатку, мало на манипуляцию).
- Counter-консистентность: unread-счётчики не должны ломаться при delete —
  проверено в session 36.

## Альтернативы (отклонены)

1. **Полный запрет (как ADR-004)** — отклонено: UX хуже привычного TG,
   пользователи всё равно ожидают edit опечаток.
2. **Hard-delete** — отклонено: теряем audit-trail, ломает counter-логику.
3. **Edit без time-окна** — отклонено: позволяет «переписать» спор задним
   числом, риск для order-thread разбирательств.
