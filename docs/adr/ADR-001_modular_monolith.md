# ADR-001 — Modular Monolith вместо микросервисов

**Дата:** 2026-03-21
**Статус:** Accepted

## Контекст

Проект стартует с командой 3 человека. Нужен backend, который:
- быстро разрабатывается,
- легко деплоится,
- поддерживает транзакционную целостность (заказы, stock, чат),
- не требует сложной инфраструктуры (service mesh, distributed tracing, etc.).

## Решение

Использовать **domain-oriented modular monolith** на NestJS.

Единый процесс с чёткими границами между модулями:
- каждый домен в своём module
- cross-module вызовы только через явные interfaces/services, не через прямой DB access
- подготовлено к extractions в сервисы если потребуется

## Причины

- Маленькая команда — нет смысла в overhead микросервисов
- Транзакционная целостность тривиальна в монолите
- Быстрый DX — один репозиторий, один деплой, единый логер
- Доменная модель ещё будет эволюционировать — рано фиксировать сервисные границы

## Последствия

- Модули должны иметь явные boundaries (нельзя писать `prisma.order.findMany` прямо в `chat.service`)
- При росте нагрузки сначала scaling монолита (вертикальный / горизонтальный), потом extraction
- Chat gateway (Socket.IO) живёт в том же процессе — приемлемо для MVP

## Альтернативы которые были рассмотрены

- **Микросервисы с gRPC** — преждевременно, добавляет distributed systems problems без benefit-а
- **Serverless** — плохо для WebSocket и long-running jobs
