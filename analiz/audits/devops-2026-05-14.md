## 🔴 P0 — Compliance

### #1 ✅ `.env.example` содержал Eskiz переменные — нарушение CLAUDE.md key rule

**Был:**
```
# OTP — Eskiz.uz
ESKIZ_EMAIL=
ESKIZ_PASSWORD=
ESKIZ_FROM=savdo
```

**Проблема:** CLAUDE.md key rule #0: «ESKIZ.UZ ЗАПРЕЩЁН — никакого SMS, никакого Eskiz, никакого Playmobile. Нарушение = откат PR.» В `.env.example` это создаёт впечатление что Eskiz может быть включён — кто-то увидит и попытается настроить.

**Fix:** Полностью переписан `.env.example`:
- Удалены 3 Eskiz переменные
- Добавлен явный комментарий «❌ SMS/Eskiz/Playmobile ЗАПРЕЩЕНЫ»
- Добавлены отсутствующие переменные новых задач: `CART_ABANDONMENT_ENABLED`, `WISHLIST_NOTIFY_ENABLED`, `LOG_LEVEL`, `ERROR_REPORTER_ENABLED`, `VITE_YANDEX_MAPS_API_KEY`, `STORAGE_REGION`
- Документация структуры: указано что есть per-app `.env.example` в `apps/<name>/`

## 🟡 P1 — Deployment correctness

### #2 ✅ Корневой `railway.toml` дублировал `apps/tma/railway.toml`

**Был:** в `/railway.toml`:
```toml
dockerfilePath = "apps/tma/Dockerfile"
watchPatterns = ["apps/tma/**"]
```

**Проблема:** main-ветка Railway service, если он создавался от этого корневого файла, авто-собирался как TMA. Согласно branch-deploy-map (`main → постоянно мерджится, не деплоится; api → service api; tma → service tma; admin → service admin; web-buyer → service web-buyer; web-seller → service web-seller`) корневой `railway.toml` создаёт неоднозначность.

**Fix:** Удалён через `git rm railway.toml`. Branch-watch в Railway работает через per-app railway.toml.

### #3 ✅ admin Dockerfile использовал `npm`, а проект — `pnpm` (workspace)

**Был:**
```dockerfile
COPY apps/admin/package.json ./
COPY apps/admin/package-lock.json* ./
RUN npm install
COPY apps/admin/ .
RUN npm run build
```

**Проблема:**
1. Admin собирался изолированно от monorepo — нет доступа к `packages/types` и `packages/ui` через workspace.
2. `package-lock.json*` (с `*`) подразумевает что lockfile может отсутствовать — но в monorepo есть `pnpm-lock.yaml` в корне, npm его не понимает.
3. Когда подключим `@savdo/ui` (после `DESIGN-PHONE-INPUT-PACKAGE-001`) admin не сможет его получить.

**Fix:** Переписан Dockerfile под `pnpm@9.0.0` workspace-pattern (как у api/tma). Build context = repo root, копируется `pnpm-lock.yaml + pnpm-workspace.yaml + packages/types + packages/ui + apps/admin/`. Команда `pnpm --filter admin build`.

### #4 ✅ `cd-backend.yml` создавал конфликт с Railway branch-watch

**Был:**
```yaml
on:
  workflow_run:
    workflows: ["CI — Backend"]
    branches: [main]
steps:
  - run: railway up --service api --detach
```

**Проблема:** workflow триггерится **после** успешного CI на main, и пытается деплоить main commit в Railway service `api`. Но Railway service `api` подписан на **ветку api**, не main. Это создавало:
- Двойной деплой (Railway watch + GitHub Actions CLI)
- Confusion при rollback (откуда деплоено?)
- Health check после CLI deploy мог фейлиться потому что Railway ещё не finalized деплой

**Fix:** Переименован в `[manual only]`, триггер изменён на `workflow_dispatch` (только запуск из GitHub UI). Auto-trigger удалён. Notifications (Telegram alerts on deploy success/fail) — рекомендовано настроить через Railway → Webhooks → Telegram (внешний канал, без GitHub Actions secrets).

## 🟢 P3 — Hygiene

### #5 ✅ `apps/api/Dockerfile` имел legacy cache-bust hack

**Был:**
```dockerfile
# cache-bust: forces rebuild so dist is always fresh
RUN echo "cache-bust-2026-04-01" > /dev/null
```

**Проблема:** Hack от 2026-04-01 — больше месяца назад. Сейчас Railway корректно реагирует на изменения через `watchPatterns`, hack не нужен. Дополнительно: hack засоряет Docker image слой (хоть и пустой).

**Fix:** Удалён.

### #6 ✅ `watchPatterns` TMA + admin не включали `packages/types/**`

**Был (TMA):** `watchPatterns = ["apps/tma/**"]`
**Был (admin):** `watchPatterns = ["apps/admin/**"]`

**Проблема:** Изменения в `packages/types/src/api/*` (DTO contracts) не триггерят rebuild TMA / admin. При изменении контракта API → нужно вручную пушить пустой коммит чтобы Railway пересобрал.

**Fix:** Добавлены `"packages/types/**", "packages/ui/**"` в обоих файлах.

### #7 ✅ Корневой `.env.example` устарел — нет новых feature flags

Помимо Eskiz (P0), отсутствовали:
- `CART_ABANDONMENT_ENABLED` (Wave 10)
- `WISHLIST_NOTIFY_ENABLED` (Wave 14)
- `LOG_LEVEL` (Wave 17)
- `ERROR_REPORTER_ENABLED` (Wave 18)
- `VITE_YANDEX_MAPS_API_KEY` (Wave 14)
- `STORAGE_REGION`

**Fix:** Все добавлены в переписанный `.env.example`.

## 🟢 Не исправлено (low-risk, зоны Азима, intentional)

### #8 `apps/web-buyer/railway.toml` хардкодит API URL

```toml
[build.args]
NEXT_PUBLIC_API_URL = "https://savdo-api-production.up.railway.app"
```

**Контекст:** web-seller использует Railway service-ref `${{api.RAILWAY_PUBLIC_DOMAIN}}`. Web-buyer — хардкод.

**Почему не трогаю:** apps/web-* — зона Азима. Если Polat поправит, может конфликт с активной работой Азима. Запишу в action items для Азима после `WEB-AUDIT-SYNC-IDEOLOGY-001`.

### #9 TMA Dockerfile `ARG VITE_API_URL=<hardcoded>`

```dockerfile
ARG VITE_API_URL=https://savdo-api-production.up.railway.app
```

**Почему не трогаю:** default value служит fallback'ом — если Railway не передаст `build.args.VITE_API_URL`, build всё равно соберётся с production URL. Это **намеренный safety net**, не bug. Можно было бы убрать в пользу error при отсутствии, но это breaking change для local builds.

### #10 Healthcheck для frontend = `/` (не `/healthz`)

Все frontend'ы (admin/tma/web-buyer/web-seller) healthcheck = `/`. Возвращает HTML index.html.

**Почему не трогаю:** работает корректно — nginx/Next.js отдают 200 на корень. `/healthz` endpoint требовал бы дополнительной route в Next.js (для web-*) или nginx conf change (для admin/tma). Сейчас работает достаточно надёжно.
