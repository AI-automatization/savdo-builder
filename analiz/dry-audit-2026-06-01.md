## P1 — критичные дубли (высокий impact, легко исправить)

### [DUP-001] 8 admin store/user status-change use-cases — копия друг друга

- **Где:**
  - `apps/api/src/modules/admin/use-cases/suspend-store.use-case.ts`
  - `…/unsuspend-store.use-case.ts`
  - `…/archive-store.use-case.ts`
  - `…/reject-store.use-case.ts`
  - `…/approve-store.use-case.ts`
  - `…/unapprove-store.use-case.ts`
  - `…/suspend-user.use-case.ts`
  - `…/unsuspend-user.use-case.ts`
- **Что:** идентичный скелет `findStoreById → if (!entity) throw NOT_FOUND → if (entity.status === X) throw CONFLICT → updateStatus → writeAuditLog`. Разница только в литералах статусов и action-кодах. Каждый файл ~35 строк, дельта между ними — 3-5 строк.
- **Почему важно:** при добавлении 9-го перехода (например, RESTORE_FROM_ARCHIVE) опять копи-паста. Любая корректировка инварианта AuditLog (новые поля payload, актор) — править в 8 местах. Тесты также копи-паста (видно по `*.spec.ts` для каждого).
- **Фикс:** generic `AdminStatusTransitionUseCase<TEntity, TStatus>` или фабрика `createStatusTransitionUseCase({ entity, find, fromStatus, toStatus, action, errorCodes })`. Spec.ts тоже table-driven.
- **Effort:** M (рефактор + переписать 8 specs).

### [DUP-002] Channel-post helpers скопированы 1:1 между preview и post

- **Где:**
  - `apps/api/src/modules/products/use-cases/preview-channel-post.use-case.ts:176-200+` (free functions `formatPrice`, `pickAttribute`, `pickSizes`, `buildProductUrl`)
  - `apps/api/src/modules/products/use-cases/post-product-to-channel.use-case.ts:133-212` (приватные методы `formatPrice`, `extractAttribute`, `extractSizes`, `buildProductUrl`, `buildContact`, `buildTemplateVariables`)
- **Что:** тот же `formatPrice` (числовое → `ru-RU` локаль + currency), тот же regex/lowercase в извлечении атрибутов, тот же `buildProductUrl` с `buyer-url || t.me/?startapp=…`. У `preview` методы — free functions, у `post` — приватные методы класса, поэтому grep не сразу подсветил.
- **Почему важно:** preview ОБЯЗАН рендерить байт-в-байт идентичное реальному посту. Сейчас они расходятся: в `post` есть `o-lcham` алиас для size, в `preview` его нет. Дрейф уже начался.
- **Фикс:** `apps/api/src/modules/products/services/channel-post-builder.service.ts` с одним `build(product, store, productUrl): TemplateVariables`. Оба use-case'а его вызывают.
- **Effort:** S (один новый сервис + удалить ~80 строк копи-пасты).

### [DUP-003] Три реализации slugify + игнорируемый SlugService

- **Где:**
  - `apps/api/src/modules/stores/services/slug.service.ts` — канонический `SlugService.generate()` (60 символов)
  - `apps/api/src/modules/admin/use-cases/admin-create-store.use-case.ts:20-28` — приватный `slugify()` (тот же regex)
  - `apps/api/src/modules/categories/repositories/store-categories.repository.ts:43-50` — приватный `toSlug()` (80 символов, fallback `'category'`)
  - `apps/admin/src/pages/CategoriesPage.tsx:21-27` — клиентский `slugify()` (другой regex, поддержка кириллицы)
- **Что:** четыре разных результата для одного входа. Особенно опасно, что admin frontend генерирует "предварительный" slug кириллицей, а backend пересоздаёт его латиницей → форма врёт пользователю.
- **Фикс:** перенести `SlugService` в `packages/db/src/utils/slug.ts` (или `packages/types`), вызывать оттуда. Frontend импортирует ту же функцию через published `@savdo/types` (см. DUP-004).
- **Effort:** S.

## P2 — структурный copy-paste

### [DUP-004] Pagination DTO копи-паста в 25+ файлах

- **Где:** все `apps/api/src/modules/**/dto/list-*.dto.ts` (`list-stores`, `list-users`, `list-sellers`, `list-orders`, `list-inbox`, `list-cases`, `list-audit-log`, `query-events`, `list-notification-logs`, …).
- **Что:** одинаковый блок:
  ```ts
  @IsOptional() @Type(() => Number) @IsInt() @Min(1) page?: number = 1;
  @IsOptional() @Type(() => Number) @IsInt() @Min(1) @Max(100) limit?: number = 20;
  ```
- **Фикс:** `apps/api/src/common/dto/pagination.dto.ts` → `class PaginationDto { page?; limit?; }` + наследование `extends PaginationDto`. Bonus: `class-transformer` `@Expose()` + DTO разделение через `IntersectionType`.
- **Effort:** S.

### [DUP-005] Ownership-check `storeId !== sellerStoreId` в 11 use-cases

- **Где:** `products/use-cases/*` (delete/update/adjust-stock/create-variant/update-variant/delete-variant/change-product-status), `categories/use-cases/*` (update-store-category, delete-store-category), `orders/use-cases/update-order-status.use-case.ts`, `products.controller.ts`.
- **Что:** одинаковый блок `if (product.storeId !== storeId) throw new DomainException(FORBIDDEN, 'Product does not belong to your store', FORBIDDEN);`
- **Фикс:** хелпер `assertProductOwnership(product, storeId)` в `apps/api/src/modules/products/guards/` либо метод-репо `findByIdAndStore(id, storeId)` который уже бросает FORBIDDEN/NOT_FOUND.
- **Effort:** S.

### [DUP-006] Pagination math `page/limit/skip` в 8 репозиториях

- **Где:** `admin.repository.ts` (8 раз), `orders.repository.ts` (4), `notification.repository.ts` (2), `analytics.repository.ts` (1), `products.repository.ts`, `moderation.repository.ts`, `reviews.repository.ts`.
- **Что:**
  ```ts
  const page = filters.page ?? 1;
  const limit = Math.min(filters.limit ?? 20, 100);
  const skip = (page - 1) * limit;
  ```
- **Фикс:** `function toPrismaPagination(input): { skip, take }` в `apps/api/src/common/utils/pagination.ts`. Заодно вернуть `meta: { page, limit, total }` через общий хелпер.
- **Effort:** S.

## P3 — стилистические дубли (низкий приоритет)

### [DUP-007] 4 копии `formatDate` в admin страницах

- **Где:** `BroadcastPage.tsx:17`, `AuditLogsPage.tsx:34`, `ChatsPage.tsx:28`, `ReportsPage.tsx:21`.
- **Что:** все вызывают `new Date(iso).toLocaleString(locale, { day: '2-digit', month: '2-digit', …})` с минимальными вариациями (year 'numeric' vs '2-digit', null-handling).
- **Фикс:** `apps/admin/src/lib/format.ts` → `formatAdminDate(iso, locale, opts?)`.
- **Effort:** XS.

### [DUP-008] `packages/types` фактически не используется

- **Где:** во всём `apps/admin/src` только 1 импорт `@savdo/types` (`SubscriptionsPage.tsx`). В `apps/api/src` — **0 импортов**. `packages/types/src/api/products.ts` содержит детально документированный `ProductListItem` (113 строк), но никто его не импортирует.
- **Что:** в `SubscriptionsPage.tsx:9-37` явно сказано: «Локальные типы зеркалят `packages/types/src/api/subscriptions.ts` — admin не подключает `@savdo/types` как npm-зависимость, поэтому копируем форму DTO». Это анти-паттерн в чистом виде — два источника правды.
- **Фикс:** добавить `"@savdo/types": "workspace:*"` в `apps/admin/package.json` и `apps/api/package.json`, прогнать import-replace по локальным `interface ProductListItem`/`AdminSubscriptionListItem`/`Notification`.
- **Effort:** M (sweep + проверить tsconfig paths).

### [DUP-009] Маленькие passthrough use-cases (~10 строк каждый)

- **Где:** `mark-inbox-read.use-case.ts`, `delete-inbox-notification.use-case.ts`, `mark-all-inbox-read.use-case.ts`, `remove-from-wishlist.use-case.ts`.
- **Что:** use-case делает один вызов в repo, никакой логики/проверок. Архитектурно ок, но boilerplate × 4. Не критично — оставить для консистентности слоёв либо вызывать репо напрямую из controller.
- **Фикс:** не рефакторить, добавить ADR «когда use-case оправдан».
- **Effort:** N/A.

## Ложные срабатывания (проверил — НЕ дубли)

- `SellersRepository.findById` vs `UsersRepository.findById` — выглядят одинаково, но include-relations разные (`{ user, store }` vs `{ seller, buyer, admin }`). Это разные агрегаты, не дубль.
- `apps/admin/src/lib/api.ts` — централизованный fetch-wrapper уже есть и используется во всех 19 страницах через `useFetch`. Дублей API-клиентов нет (грозили мифом).
- `cn(...inputs)` в `apps/admin/src/lib/utils.ts` — единственная реализация, ОК.
- Маленькие inbox use-cases (DUP-009) — формально passthrough, но это согласованная архитектура NestJS Clean, оставляем.
